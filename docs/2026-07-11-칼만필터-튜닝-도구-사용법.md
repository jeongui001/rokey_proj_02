# 칼만필터 + P 제어식(w·v̂_tool + Kp·e) 오프라인 튜닝 도구 사용법

이 문서는 방금 `main`에 병합된 3개 도구(`tool_track_logger`, `kalman_replay`,
`simulate_conveyor`)를 실제로 어떻게 돌려서 `servo:` 파라미터를 튜닝하는지
설명한다. 설계 배경은
`docs/superpowers/specs/2026-07-11-kalman-servo-control-tuning-design.md`
(로컬 파일, `.gitignore`로 git에는 없음) 참고.

## 0. 사전 준비

워크스페이스가 이미 source된 상태에서 아래 명령이 바로 동작하는지 확인:

```bash
cd ~/rokey_proj_02/src/robot_control
python3 -c "import robot_control.tools.kalman_replay"
```

에러 없이 넘어가면 준비 완료. `tool_track_logger`만 실제 ROS2(`rclpy`,
`handover_interfaces`)가 필요하고, `kalman_replay`/`simulate_conveyor`는
순수 Python이라 ROS 노드를 안 띄워도 CLI로 바로 실행된다.

## 전체 흐름 한눈에 보기

```
[1] 렌치를 카메라 앞에 두고 tool_track_logger 실행
        ↓ (CSV: stamp_s, recv_monotonic_s, x, y, z, depth_valid)
[2] 그 CSV로 r_xy/r_z, t_lost를 numpy로 계산
        ↓
[3] kalman_replay --target kalman 으로 r_xy/r_z 게이트 확인
        ↓
[4] 컨베이어(이동+방향전환) 로그로 kalman_replay --target kalman 스윕
        → q_pos/q_vel 확정
        ↓
[5] 같은 로그로 kalman_replay --target servo 스윕
        → innov_low/innov_high/w_alpha/t_lost 게이트 확인
        ↓
[6] robot_control_params.yaml의 servo: 섹션에 확정값 반영
        ↓
[7] simulate_conveyor --kp-xy 로 사전점검 후 실기 폐루프로 kp_xy 최종 확정
```

`dt_latency`(2단계)만 이 도구들로 안 되고 실기 스텝 실험이 필요하다 (아래 6절).

## 1. 데이터 수집: `tool_track_logger`

`/vision/tool_track`을 구독해서 CSV로 저장하는 ROS2 노드. `vision_node`가
떠 있고 렌치가 카메라 시야 안에 있어야 한다.

```bash
# 정지 렌치 로그 (r_xy/r_z, t_lost 산출용, 1/3단계)
ros2 run robot_control tool_track_logger --out /tmp/static_track.csv

# (ros2 run이 새 entry_point를 못 찾으면 워크스페이스를 다시 빌드해야 함:
#  colcon build --packages-select robot_control 후 install/setup.bash 재source.
#  또는 그냥 직접 실행: python3 -m robot_control.tools.tool_track_logger --out /tmp/static_track.csv)
```

- 렌치를 시야에 정지시켜두고 200~500프레임 정도 모일 때까지(수 초) 기다린 뒤
  `Ctrl+C`로 종료. 종료 시 `N개 프레임 저장 완료: ...` 메시지가 뜬다.
- `--topic`으로 다른 토픽 이름을 줄 수 있음(기본 `/vision/tool_track`).
- 이동(컨베이어) 로그가 필요한 4/5단계에서는 같은 명령으로 렌치가 벨트 위에서
  등속 이동 + 방향전환하는 구간을 포함해서 몇 초간 수집(`/tmp/moving_track.csv`
  같은 이름으로 저장).

출력 CSV 형식:

```
stamp_s,recv_monotonic_s,x,y,z,depth_valid
10.500,123.456,0.10,0.20,0.30,True
```

- `stamp_s`: ToolTrack 메시지의 `header.stamp` (카메라 촬영 시각, 초)
- `recv_monotonic_s`: 로거가 메시지를 받은 시각(`time.monotonic()`) — 3단계
  t_lost 산출과 `kalman_replay --target servo`의 시간 재생에 쓰인다
- `x,y,z`: base_link 기준 위치(m)
- `depth_valid`: depth 관측 유효 여부

## 2. r_xy / r_z / t_lost 산출 (전용 도구 없음 — numpy 한 줄)

정지 로그(`/tmp/static_track.csv`)를 `kalman_replay`의 `read_track_csv()`로
읽어서 바로 계산한다. 전용 스크립트를 안 만든 이유는 이게 numpy 3줄짜리라서다.

```bash
python3 -c "
import numpy as np
from robot_control.tools.kalman_replay import read_track_csv

rows = read_track_csv('/tmp/static_track.csv')
xs = np.array([r.x for r in rows])
ys = np.array([r.y for r in rows])
zs = np.array([r.z for r in rows])
r_xy = float((xs.var() + ys.var()) / 2)
r_z = float(zs.var())
print(f'r_xy={r_xy:.6g}  r_z={r_z:.6g}')

gaps = np.diff([r.recv_monotonic_s for r in rows])
t_lost = float(np.percentile(gaps, 99)) * 1.5  # p99 + 50% 안전마진(값은 상황 봐서 조정)
print(f't_lost={t_lost:.4f}s  (max gap={gaps.max():.4f}s)')
"
```

- `r_xy`, `r_z`가 지나치게 작게 나오면(예: `1e-8` 이하) 실제 카메라 노이즈보다
  낙관적일 수 있다 — 렌치를 살짝 흔든 상태로 재수집해서 비교해보는 것도 방법.
- `t_lost`는 "정상적인 프레임 간격"과 "진짜 유실"을 가르는 값이라, 계산된 값을
  그대로 쓰기보다 3단계 게이트(아래)로 오탐/누락 여부를 확인 후 조정.

## 3. `kalman_replay --target kalman`: 칼만필터 단독 검증 (1/4단계)

칼만필터(`KalmanXYZV`)만 떼어서 CSV를 재생한다. `ServoLoop`의 `w`/신뢰가중치
로직 없이 필터 자체의 수렴만 본다.

```bash
python3 -m robot_control.tools.kalman_replay \
  --in /tmp/static_track.csv --out /tmp/static_kalman_out.csv \
  --target kalman \
  --r-xy 1e-4 --r-z 1e-4 --q-pos 1e-4 --q-vel 1e-2 --p0-vel-reset 1.0
```

출력 CSV 컬럼: `stamp_s, innovation_xy_m, x, y, z, vx, vy` (첫 행은
`innovation_xy_m`이 빈칸 — 초기화만 하고 아직 갱신 전이라서 그렇다).

**1단계 게이트(정지 로그로 확인):** 마지막 몇 행의 `x, y`가 로그의 실제
평균값과 몇 mm 이내로 수렴하는지 확인. 예:

```bash
python3 -c "
import csv
with open('/tmp/static_kalman_out.csv') as f:
    rows = list(csv.DictReader(f))
last = rows[-1]
print('마지막 추정 위치:', last['x'], last['y'], last['z'])
"
```
발산하거나(값이 요동) 너무 느리게 수렴하면 `--r-xy`/`--r-z`를 낮추거나
`--q-pos`를 조정해서 재실행 — 같은 로그에 여러 값을 반복 실행해 비교하면 된다.

**4단계 게이트(이동 로그로 확인):** `/tmp/moving_track.csv`로 같은 명령을
`--q-pos`/`--q-vel` 후보값을 바꿔가며 여러 번 돌리고, 출력 CSV의
`innovation_xy_m` 컬럼을 비교:
- 등속 구간에서 `innovation_xy_m`이 낮고 안정적인가
- 방향전환 시점(`x`가 반전하는 부근) 이후 몇 행 만에 `vx`가 새 방향값으로
  재수렴하는가 (너무 오래 걸리면 `q_vel`을 올려서 재수렴을 빠르게)

`--q-pos`가 너무 크면 정상 구간에서도 `innovation_xy_m`이 들쑥날쑥해지고,
너무 작으면 방향전환 후 재수렴이 느려진다 — 두 극단 사이에서 로그를 눈으로
비교하며 값을 좁혀간다.

## 4. `kalman_replay --target servo`: w / innovation / t_lost 검증 (3/5단계)

`ServoLoop` 전체(칼만필터 + 신뢰가중치 `w` + abort 판정)를 재생한다.
`step()`은 실측 TCP가 없어서 호출하지 않으므로 `e_xy_norm`/`z_gap`/
`should_close`/`'diverging'`은 이 도구로 검증할 수 없다 — 오직
`w`/`innovation`/`tracking_lost`/`timeout`만 확인 가능.

```bash
python3 -m robot_control.tools.kalman_replay \
  --in /tmp/moving_track.csv --out /tmp/moving_servo_out.csv \
  --target servo \
  --r-xy 1e-4 --r-z 1e-4 --q-pos 1e-4 --q-vel 1e-2 --p0-vel-reset 1.0 \
  --innov-low 0.010 --innov-high 0.040 --w-alpha 0.3 \
  --dt-latency 0.05 --t-lost 0.3 --timeout 10.0
```

출력 CSV 컬럼: `stamp_s, w, w_target, innovation_xy_m, depth_valid,
abort_reason` (`abort_reason`이 빈칸이면 정상).

**5단계 게이트(innov_low/innov_high/w_alpha):**
- 정상(등속) 구간에서 `w`가 1 근처로 안정적인가 (아래 예시 출력에서
  `w=0.983` 수준이면 정상)
- 방향전환 시점에서 `w`가 빠르게 떨어졌다가(0 근처) 재수렴하는가 —
  너무 안 떨어지면 `--innov-high`를 낮추고, 너무 자주/오래 떨어지면
  `--innov-low`를 낮추거나 `--w-alpha`로 스무딩을 조정
- `w`가 채터링(짧은 주기로 위아래 진동)하면 `--w-alpha`를 낮춤(더 느리게 반응)

**3단계 게이트(t_lost 오탐 확인):** `abort_reason` 컬럼에서
- 정상 추적 로그(프레임 드롭 없음)를 재생했을 때 `'tracking_lost'`가 한 번도
  안 나오면 통과. 나오면 `--t-lost`를 늘림.
- 로그의 전체 시간 길이가 `--timeout`(기본 10초)을 넘으면 그 이후 행은
  `tracking_lost` 여부와 무관하게 `'timeout'`으로 찍힌다 — 긴 로그로
  `tracking_lost` 오탐만 확인하고 싶으면 `--timeout`을 로그 길이보다 크게
  줘야 한다(예: `--timeout 60`).

### 참고: 실제 실행 결과 예시 (합성 데이터, 등속 40프레임)

```
stamp_s,w,w_target,innovation_xy_m,depth_valid,abort_reason
0.0,0.0,,,True,
0.02,0.3,1.0,0.002,True,
0.04,0.51,1.0,0.001999,True,
```

첫 행은 초기화만 하고 `w_target`이 아직 없어 `w=0.0`, 이후 `w_alpha=0.3`
저역통과로 서서히 1에 수렴해가는 걸 볼 수 있다.

## 6. `dt_latency` — 도구로 안 되는 유일한 값 (2단계)

로그 재생만으로는 카메라 캡처~추론~통신~로봇 반응까지의 실제 물리 지연을
알 수 없다. 실기에서:

1. 정지해 있던 렌치를 순간적으로 빠르게 이동시킨다.
2. 그 순간의 `ToolTrack.header.stamp`(로거 CSV의 `stamp_s`)부터 로봇이 실제
   속도로 반응하기 시작하는 시점까지 시간차를 잰다.
3. 여러 번 반복해서 p95 값을 `dt_latency`로 채택, yaml에 반영.

## 7. `simulate_conveyor --kp-xy`: kp_xy 사전점검 + 실기 확정 (6단계)

이 도구는 위 1~6단계와 무관하게 이미 있던 합성(노이즈 없는) 시뮬레이션이다.
실기 전에 대략적인 `kp_xy` 후보를 값싸게 좁히는 용도.

```bash
python3 -m robot_control.tools.simulate_conveyor --kp-xy 0.8 1.0 1.2 1.5 2.0
```

출력 예:
```
kp_xy=0.8 constant: avg_w=0.983 avg_e_xy=0.0942m
kp_xy=1.0 constant: avg_w=0.983 avg_e_xy=0.0863m
kp_xy=1.2 constant: avg_w=0.983 avg_e_xy=0.0820m
...
```

`constant`/`long_reversal`/`short_oscillation` 3개 시나리오 각각에 대해
`avg_e_xy`(평균 추종 오차)가 가장 작은 `kp_xy`가 시작값 후보다. 단, 이
시뮬레이션은 센서 노이즈·실제 로봇 동역학이 전혀 없는 이상화 모델이라
**참고용일 뿐**이다.

이 시작값으로 실제 `servo_pick`을 실기에서 돌려 스텝 응답(초기 오차 →
수렴 과정)을 관찰(`debug.log_servo_decisions` 로그 또는
`servo_loop.debug_snapshot()`의 `e_xy_norm_m` 추이):
- 오버슈트/진동이 보이면 `kp_xy`를 낮춤
- 수렴이 느리면 `kp_xy`를 높임
- `eps_grasp` 이내로 안정적으로 수렴하면 확정, yaml에 반영

## 명령어 치트시트

```bash
# 1. 정지 로그 수집
ros2 run robot_control tool_track_logger --out /tmp/static_track.csv

# 1. 이동(컨베이어) 로그 수집
ros2 run robot_control tool_track_logger --out /tmp/moving_track.csv

# 2. r_xy/r_z/t_lost 계산 (numpy 한 줄, 2절 스크립트 참고)

# 3/4. 칼만필터 단독 재생
python3 -m robot_control.tools.kalman_replay --in <csv> --out <out.csv> \
  --target kalman --r-xy <값> --r-z <값> --q-pos <값> --q-vel <값>

# 3/5. ServoLoop 전체 재생 (w/innovation/t_lost)
python3 -m robot_control.tools.kalman_replay --in <csv> --out <out.csv> \
  --target servo --r-xy <값> --r-z <값> --q-pos <값> --q-vel <값> \
  --innov-low <값> --innov-high <값> --w-alpha <값> \
  --dt-latency <값> --t-lost <값> --timeout <로그길이보다 크게>

# 6. kp_xy 사전점검
python3 -m robot_control.tools.simulate_conveyor --kp-xy 0.8 1.0 1.2 1.5 2.0

# 전체 CLI 옵션 확인
python3 -m robot_control.tools.kalman_replay --help
```

## 트러블슈팅

- `ros2 run robot_control tool_track_logger`가 "실행 파일을 찾을 수 없음"
  이라 뜨면 → `colcon build --packages-select robot_control` 후
  `install/setup.bash` 재source. 급하면 `python3 -m
  robot_control.tools.tool_track_logger --out ...`로 직접 실행해도 동일하게
  동작한다.
- `kalman_replay`/`simulate_conveyor` 실행 시 `ModuleNotFoundError:
  robot_control` → 워크스페이스가 source 안 된 상태. `source
  install/setup.bash` 먼저.
- `--in` CSV의 헤더가 `stamp_s,recv_monotonic_s,x,y,z,depth_valid`와 다르면
  `KeyError`가 난다 — `tool_track_logger` 출력을 그대로 쓰면 문제없음.
