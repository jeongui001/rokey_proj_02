# 공구 전달 로봇 시스템 설계 문서

ROS2 기반 음성 명령 공구 전달 시스템 — 노드 명세, 상태 머신, 인터페이스, 시퀀스
(파지 전략: 비주얼 서보잉 — 하강 중 연속 추적)
멀티 패키지 구조 (Multi-package)로 제작할 것

## 0. 확정된 설계 결정 요약

| 항목 | 결정 |
|---|---|
| 로봇팔 | Doosan M/A 시리즈. 서보 파지 구간은 비-RT `speedl_stream`(고주기 목표 스트리밍, RT 세션 불필요)으로 제어(2026-07-07부터 - 원래는 RT 세션이었으나 실측으로 비-RT만으로도 충분함을 확인해 전환). 명령이 끊겨도 로봇이 스스로 멈추지 않아 애플리케이션 워치독(`SpeedlWatchdog`, 데드맨스위치 스레드)이 자동 정지를 대신한다 |
| 그리퍼 | OnRobot RG2, Compute Box에 Modbus TCP — **robot_control 노드에 통합** (별도 노드 없음) |
| 카메라 | 손목 RealSense(eye-in-hand) 단독. 고정 RGB 카메라 없음. 저해상도·고프레임(424×240@60fps 시작점)으로 MinZ와 비전 지연을 동시에 최소화 |
| 컨베이어 | 양방향 왕복, 로봇 시스템에서의 제어 없음(노드 없음) — 모터 속도·가속도를 직접 조절해 ①한방향 등속 ②긴 주기 왕복 ③짧은 주기 진동 3가지 프로파일로 운용/시연. 파지는 감시 자세의 RealSense 시야 구간에서만 수행 |
| 파지 전략 | **비주얼 서보잉(PBVS)**: 하강 중에도 RealSense 추적을 계속 소비해 목표를 매 주기 갱신. robot_control 내장 서보 루프가 필터 추정치로 실시간 추종하고, 상태 기반(오차·안정성) 판정으로 폐합. 시간 정합과 지연 보상(Δt_lat)이 핵심 난관. **servo_pick 전용** — handover_approach(손 접근)는 정밀 추종이 필요 없어 movel 기반 단발성 이동으로 별도 구현(현재 TODO 스텁, 8절 참고) |
| 명령 처리 | 로컬 Whisper STT → `task_manager`의 키워드 기반 파서(`command_parser`)로 task_type/named_target 추출 — **LLM API 미사용**(초기 설계는 LLM이었으나 실제 구현은 동기 키워드 매칭으로 대체됨) |
| UI | PyQt 데스크톱 앱(`operator_gui`) ↔ ROS2. rclpy로 직접 접속(백그라운드 스레드에서 spin), 카메라 디버그 영상(CompressedImage)도 표시. 초안은 브라우저+rosbridge/roslibpy였으나, UI가 PyQt로 확정된 뒤 rclpy 직접 연동으로 변경 |
| 파지/당김 감지 | Doosan 조인트 내장 토크(외부 토크) + RG2 grip_detected + 파지 폭 삼중 확인 |

### 1.4 vision_node — RealSense 전담

| | |
|---|---|
| 역할 | 저해상도·고프레임 파이프라인. 모드에 따라 공구 추적 또는 손 추적. object_detection(팀원3)의 검출 결과 수신 → 추적 → 3D 복원 → base_link 변환 → 스트리밍 |

| TRACK_TOOL | object_detection이 저해상도 입력(YOLO 416 또는 320)으로 검출한 bbox를 받아 프레임 간 추적해 pose·velocity·yaw 추정, `approaching` 판정. depth가 MinZ 이내 근접 등으로 무효화되면 `depth_valid=false`로 표시하고, 이 구간에서도 RGB 추적으로 x·y는 계속 갱신(z는 마지막 유효값 고정) |
| TRACK_HAND | 손 검출(MediaPipe 등) → 전달 위치 미세 조정용 손 자세 |
| 비고 | 모든 메시지에 관측 시각(header.stamp) 필수 — 시간 정합·지연 보상의 기준. 3D 복원용 TF는 "지금"이 아니라 "이미지가 촬영된 시각"의 flange pose로 조회(`tf_buffer.lookup_transform(..., 이미지 header.stamp)`). 비전 PC와 로봇 제어 PC가 분리되어 있으면 chrony 시계 동기화 필수 |

### 1.5 robot_control — 실행자 + 서보 제어 루프 내장

| | |
|---|---|
| 역할 | RobotTask 액션 서버. Doosan 모션 서비스(정적 이동) + 비-RT `speedl_stream`(서보 파지 전용) 호출 + RG2 그리퍼 제어(Modbus TCP 내장) + 토크 감시 |
| 액션 서버 | `RobotTask` — move_named / release_and_retry / servo_pick / handover_hold / handover_approach (**move_pose·place_down은 정리되어 더 이상 지원하지 않음** - 미지원 task_type으로 거부됨) |
| 구독 | Doosan 드라이버 로봇 상태(외부 토크 포함, 최대 주기로 발행 요청), `/vision/tool_track` (servo_pick 진행 중 직접 구독) |
| 퍼블리시 | `/gripper/state` (GripperState, 상시), `/robot/fault` (protective stop 등) |
| servo_pick | goal 수신 시 `/vision/tool_track`을 직접 구독해 필터(칼만)로 위치·속도 추정 → PBVS 제어 법칙으로 매 주기 목표 추종(2절) → 비-RT `speedl_stream`으로 속도 명령 연속 발행(`SpeedlWatchdog`이 명령 두절 시 자동 정지 대체) → 상태 기반 판정 성립 시 RG2 폐합(공구별 폭·힘) → 즉시 들어올림 → 서비스 모션 모드로 복귀 → result 반환 (※ 토크 기반 페이로드 추정은 미구현 - `measured_payload_kg` 필드 자체가 인터페이스에 없음, 4.5절 참고) |
| handover_approach | handover_safe 도착 후 작업자 손 근처(stop_distance_m 이내)까지 접근 - **정밀 추종이 불필요해 movel 기반 단발성 이동으로 구현 예정**(2026-07-07 기준 게이트 체크만 있고 본문은 TODO 스텁, 항상 실패 반환) |
| handover_hold | 컴플라이언스 모드 on + 외부 토크 임계 감시 → 당김 감지 시 RG2 개방, result 반환 |
| 내부 구조 | RG2 제어는 `rg2_client.py` 클래스로 분리. 서보 제어 루프는 `servo_loop.py`(ServoLoop, servo_pick 전용), 워치독은 `speedl_watchdog.py`(SpeedlWatchdog)로 분리 |
| RG2 통신 재시도 (2026-07-08 추가) | Modbus 소켓 타임아웃이 1초로 하드코딩돼 있어 네트워크가 잠깐 느려지기만 해도 COMMUNICATION_ERROR로 이어지던 문제를 완화 — `rg2.connect_timeout_s`(기본 2.0s)로 파라미터화하고, `_run_command_with_retry`가 COMMUNICATION_ERROR에 한해서만(같은 목표를 다시 보내는 멱등 재시도라 안전) `rg2.communication_retry_count`(기본 2회)/`rg2.communication_retry_backoff_s`(기본 0.5s)로 자동 재시도. CANCELED/FAULT 등 다른 상태는 재시도하지 않음 |
| 설정 파일 | `robot_control_params.yaml`(기본값) + `robot_control_calibration_params.yaml`(하드웨어 종속 실측 캘리브레이션 override, 2026-07-08 추가) — 후자는 launch 시 `params_file`로 전자와 함께 넘기거나 명령행 `-p` 오버라이드로 사용. 현재는 `handover_hold` 당김 감지 축/방향/임계값(2026-07-07 실기 tool_force 로그 기반)만 포함 |

**안전 감시 인프라**: `SafetyMonitor`가 NORMAL/PROTECTIVE_STOP/EMERGENCY_STOP/FAULT
4단계 안전 상태를 관리하고, `/robot/recover` 서비스로 복구를 처리한다. `DoosanDriver`가
dsr_msgs2 서비스(모션·상태 조회·`speedl_stream` 발행·컴플라이언스)를 실제로
감싸며(RT 세션 관련 서비스는 2026-07-07 제거됨), 안전 상태가
NORMAL이 아니면 진행 중인 이동을 취소한다. `DrflForceMonitor`는 DRFL 라이브러리에
ctypes로 직접 연결해 로봇이 이동 중(MOVING)에도 100Hz로 외력을 감지하는 별도
스레드로, ROS2 서비스 폴링(정지 상태 전용)과 별개의 보조 안전 레이어다.

## 2. 서보 제어 루프 설계 (robot_control 내부)

task_manager가 좌표를 조립하고 robot_control은 순수 실행만 한다는 원칙에 이 구간만 예외가 있다.
서보 루프는 지연에 민감해 task_manager를 경유할 수 없으므로, servo_pick 동안에는
robot_control이 `/vision/tool_track`을 **직접 구독**해 제어 루프를 돌린다.

```
vision ────────────────────────▶ robot_control (서보 루프, 매 주기)
              task_manager는 시작/중단/감독만 (goal/abort/result)
```

### 2.1 카메라 설정과 지연 예산

관건은 **비전 루프 지연**과 **블라인드 구간**이며, 해상도를 낮추면 둘 다 개선된다.

1. **지연 감소**: 프레임 전송·YOLO 추론 시간이 해상도에 비례해 줄어 30~60Hz 갱신이 현실화된다.
2. **MinZ 감소**: RealSense 스테레오 최소 심도 거리(MinZ)는 대략 가로 해상도에 비례한다.
   D435 기준 1280×720에서 약 28cm → 848×480 약 18~20cm → 424×240 약 10cm 수준
   (기종·프리셋별 상이, **반드시 실측**). 저해상도일수록 depth가 유효한 채로
   공구에 더 가까이 접근할 수 있어 블라인드 종단 구간이 짧아진다.

해상도 저하의 대가로 depth 노이즈와 검출 정밀도가 나빠지므로 필터(2.5절)가 필수가 된다.

| 항목 | 값 | 근거 |
|---|---|---|
| Depth 해상도/프레임 | 424×240 @ 60fps | MinZ 최소화 + 지연 최소화 |
| Color 해상도/프레임 | 424×240 @ 60fps | depth와 정렬 비용 최소화 |
| YOLO 입력 | 416 또는 320 | 추론 10~20ms(GPU) 목표 |
| 프리셋 | High Density 또는 Short Range | 근거리 fill rate 우선 |

지연 예산 (목표: 관측→명령 반영 70ms 이내):

| 구간 | 예상 |
|---|---|
| 노출 + 카메라 파이프라인 + USB | 20~35ms |
| YOLO 추론 + 3D 복원 + 필터 | 10~20ms |
| DDS 전달 + 서보 루프 반영 | 2~5ms |
| **합계 (lookahead 예측에 사용, Δt_lat)** | **약 40~60ms** |

벨트 0.10m/s 기준 40~60ms는 4~6mm에 해당하므로 보상 없이는 그대로 파지 오차가 된다.

### 2.2 Doosan 비-RT 스트리밍 인터페이스 (2026-07-07 갱신)

서비스 모션(movel: 목표 주고 완료 대기)은 도중 갱신이 불가하므로, 고주기 목표
스트리밍이 필요하다. 원래는 Doosan의 RT 세션(`ConnectRtControl`/`StartRtControl`
등)으로 구현했으나, 실측(`probe_speedl_stream.py`) 결과 **RT 세션 없이 비-RT
`speedl_stream` 토픽만으로도 동일하게 부드러운 서보잉이 가능함**을 확인해
전환했다. RT 세션 열기/닫기 코드 자체가 완전히 제거되어 있다.

다만 비-RT `speedl_stream`은 RT와 달리 **명령이 끊겨도 로봇이 스스로 멈추지
않는다**(실측 확인). 이를 보완하기 위해 `SpeedlWatchdog`(데드맨스위치, 메인
서보 루프와 별개 스레드)를 도입했다 — 메인 루프가 매 틱 `pet()`을 호출하고,
`watchdog_timeout_s`(기본 0.2초) 이내에 `pet()`이 없으면 워치독이 독립적으로
`vel=0`을 발행해 정지시킨다. 정지 후 루프가 회복되어 `pet()`이 재개되면
자동으로 재무장되어 감시를 계속한다(1회성 알람이 아니라 지속적 감시).

이 스트리밍 방식은 현재 **servo_pick 전용**이다. handover_approach는 정밀
추종이 필요 없어 movel(점대점 이동) 기반으로 별도 구현할 예정이라 이 절의
내용을 쓰지 않는다(1.5절, 8절 참고).

### 2.3 제어 법칙 (PBVS, base_link 기준)

루프 주기: 비전 갱신(30~60Hz)과 `speedl_stream` 명령 주기(`servo_pick.control_period_s`,
기본 0.01s)가 다르므로, 비전은 필터 갱신에만 쓰고 속도 명령은 필터의 예측값을
자체 주기로 내보낸다(비전 프레임 사이에는 외삽).

```
목표점  p_ref(t) = p̂_tool(t + Δt_lat) + offset_approach
오차    e = p_ref − p_tcp
명령    v_cmd = clamp(w · v̂_tool + Kp · e, v_max)   # 속도 피드포워드(가중치 w) + P 피드백, 수평(x,y) 성분
하강    z는 별도 프로파일: 수평 오차 |e_xy| < ε_descend 인 동안만 일정 속도로 하강,
        초과하면 하강 일시 정지(수평 정렬 우선)
yaw     동일하게 P 제어, 각속도 제한
```

- `p̂_tool(·)`: 필터(2.5절)의 상태로부터 `위치 + 속도 × (Δt_lat + 루프 반주기)` 외삽
- `v̂_tool`: 필터가 추정한 공구 속도(피드포워드 항). 순수 P 피드백(`Kp·e`)만으로는 등속 이동 목표에 정상상태 오차가 남으므로, 추정 속도를 명령에 직접 더해 추종 지연을 없앤다
- `w`: 피드포워드 신뢰 가중치(0~1). 필터의 innovation(2.5절)이 `innov_low` 이하면 1(등속 가정 유효, 피드포워드 전부 신뢰), `innov_high` 이상이면 0(진동 등으로 등속 가정 위반, 피드포워드 무시하고 P 피드백만 사용)으로 선형 감쇠, 잔차 노이즈로 인한 채터링 방지를 위해 `w_alpha`로 저역통과 스무딩
- Kp는 낮게 시작(0.8~1.5 /s), v_max는 벨트 속도의 2~3배면 충분

### 2.4 시간 정합 — 이 방식의 최대 함정

eye-in-hand로 팔이 움직이는 중에는, 3D 복원에 쓰는 TF 변환을 **"지금"이 아니라
"이미지가 촬영된 시각"의 flange pose**로 해야 한다.

- vision_node: `tf_buffer.lookup_transform(base, camera, 이미지 header.stamp)` 사용.
  Doosan 드라이버의 로봇 상태 퍼블리시 주기가 낮으면 TF 보간 오차가 커지므로
  상태 퍼블리시 주기를 최대로 올릴 것
- 비전 PC와 로봇 제어 PC가 분리되어 있으면 chrony 시계 동기화 필수
  (수십 ms 어긋남 = 수 mm 오차 직결)
- 검증법: 로봇을 일부러 흔들면서 정지한 공구를 추적 → base_link 좌표가
  흔들리지 않으면 정합 성공

### 2.5 필터

base_link 좌표의 공구 위치에 등속 모델 칼만 필터(또는 알파-베타 필터)를 적용.
상태 = [x, y, z, vx, vy], 저해상도 depth의 노이즈를 흡수하고 velocity를 안정적으로
추정하는 것이 목적. 벨트 **방향 전환은 등속 모델에 대한 급격한 위반**이므로,
innovation(예측-관측 잔차)이 임계를 넘으면 속도 공분산을 리셋해 빠르게 재수렴시킨다.
같은 innovation 신호를 2.3절의 피드포워드 가중치 `w` 계산에도 재사용한다 — 리셋은
"임계 초과 시 이진 트리거"이고 `w`는 "잔차 크기에 따른 연속적 신뢰도"라는 차이가 있다.

### 2.6 파지 판정과 폐합

정지 시각 예약이 아니라 **상태 기반 판정**으로 이루어진다:

```
if |e_xy| < ε_grasp (예 5mm) 가 N주기(예 5) 연속
   and z_gap < z_close (예 2cm)
   and 필터 공분산 < 임계:
       그리퍼 폐합 (RG2: 공구별 폭·힘)
```

### 2.7 종단 블라인드 구간 처리

MinZ를 10cm 수준까지 낮춰도 마지막 수 cm + 그리퍼 자체 가림은 남는다:

1. depth 무효 진입 후: z는 마지막 유효값 고정, x·y는 RGB 추적으로 계속 보정
   (`depth_valid=false` 구간)
2. RGB마저 가려지면: 필터의 마지막 속도로 짧은 탄도(ballistic) 종단 —
   구간이 1~3cm로 짧아 오차가 무시할 수준

> 참고: 해상도를 더 낮추면 MinZ가 그립 지점까지 커버되어 위 블라인드 구간 자체가
> 사라질 가능성이 있다. 다만 그리퍼 자체 가림은 해상도와 무관하게 남는 변수이므로,
> 실측(7절 1번 검증)으로 확정하기 전까지는 현행 설계(1~2번 절차, 2.6절의 z_gap 판정)를 유지한다.

### 2.8 발산·중단(abort) 조건

다음 중 하나면 서보를 중단하고 감시 자세로 상승 복귀 → task_manager에 abort 반환
→ DETECT_TRACK 재진입(토크 이상은 즉시 FAULT):

- |e_xy|가 M주기 연속 증가 (발산)
- 추적 유실(검출 끊김)이 T_lost(예 0.3s) 초과
- 공구가 방향 전환하여 시야 이탈 예상
- 서보 시간이 총 타임아웃(예 5s) 초과
- 외부 토크 이상(충돌) → 즉시 정지 + FAULT

## 3. 상태 머신 및 상태 × 노드 매트릭스 (2026-07-07 실제 구현 기준으로 갱신)

> 이 절은 최초 설계와 실제 구현이 상당히 갈라진 부분이다. 가장 큰 차이:
> **`state`(작업 진행 상태)와 `safety_state`(NORMAL/PROTECTIVE_STOP/EMERGENCY_STOP/
> FAULT/RECOVERY_REQUIRED)는 서로 독립된 필드**다(task_manager의 `task_models.py`
> `State`/`Safety` 클래스, `/task/status`에 둘 다 실려 퍼블리시됨). 아래 표의
> "실패 전이"에 적힌 FAULT는 `state`가 FAULT로 바뀌는 게 아니라 `safety_state`가
> FAULT로 격상되고 진행 중이던 goal이 취소됨을 뜻한다. 또한 **PARSING·RELEASE
> 상태는 실제로 없고**, LLM 호출도 없다(동기 키워드 파싱).

실제 `State` 전체 목록(`task_manager/task_models.py`): `IDLE`, `MOVE_TO_WATCH`,
`DETECT_TRACK`, `SERVO_PICK`, `VERIFY_GRASP`, `MOVE_SAFE`, `APPROACH_HAND`(최초
설계의 "TRACK_HAND"), `WAIT_PULL`, `HOME`, `MANUAL_MOVE`(수동 이동 모드),
`CANCELLING`(goal 취소 진행 중) — 11개.

전체 흐름(AUTO 모드, 정상 경로):

```
IDLE → MOVE_TO_WATCH → DETECT_TRACK → SERVO_PICK → VERIFY_GRASP
     → MOVE_SAFE → APPROACH_HAND → WAIT_PULL → HOME → IDLE
(모든 단계에서 result.success=false 또는 /robot/fault 수신 시
 safety_state가 FAULT로 격상되고 진행 중 goal이 취소됨 - state 자체는
 유지되며, 수동 리셋(/robot/recover 성공)으로만 state=IDLE·safety=NORMAL 복귀)
```

|---|---|---|---|---|
| IDLE | task_manager: `/user_command/text` 대기, 수신 시 `command_parser`로 동기 키워드 파싱(LLM 없음). vision: OFF | 시스템 기동 / HOME 완료 | FETCH_TOOL 파싱 성공(AUTO 모드) → MOVE_TO_WATCH. MOVE_NAMED 파싱 성공(MANUAL 모드) → MANUAL_MOVE | 파싱 실패·미지 공구·필수 파라미터(폭/힘) 미설정 → IDLE 유지, 상세 사유만 보고 |
| MOVE_TO_WATCH | robot_control: move_named(watch). task_manager: vision을 TRACK_TOOL(목표 클래스)로 전환 | IDLE에서 파싱 성공 | 이동 완료 → DETECT_TRACK(감시 타이머 시작) | 모션 실패 또는 vision TRACK_TOOL 전환 실패 → safety_state=FAULT |

| SERVO_PICK | robot_control: `/vision/tool_track` 직접 구독해 PBVS 서보 루프로 추종 → 비-RT `speedl_stream`으로 속도 명령 연속 발행(`SpeedlWatchdog`이 명령 두절 시 자동 정지) → 폐합·들어올림(2절). task_manager: goal 1회 전송 후 감독만 | 트리거 성립 | 파지·들어올림 완료(`result.success`) → VERIFY_GRASP | 메시지가 `lost`/`timeout`/`diverged`면 복구 가능 실패로 보고 → DETECT_TRACK 재진입(타이머 재시작). 그 외 실패 메시지 → safety_state=FAULT |

(손 미검출 타임아웃/`hand_detect_timeout_s` 폴백은 설계 문서에만 있고 task_manager에 구현되어 있지 않음) |
| WAIT_PULL | robot_control: handover_hold(컴플라이언스 on, 토크 임계 감시, 당김 감지 시 **그 안에서** RG2 개방까지 처리) | 손 근처 도달 | 당김 감지(`result.success`, 그리퍼 개방까지 완료된 상태) → **HOME 직행** (별도 RELEASE 상태 없음) | `wait_pull_timeout_s`(기본 60s) 경과해도 취소·재배치하지 않는다 — 로봇은 그 자리에서 계속 들고 대기하고, `wait_pull_reminder_interval_s`(기본 10s)마다 "도구를 가져가세요" 안내만 반복 퍼블리시(`/task/status.detail`). 실제 당김이 오면 그제서야 HOME으로 전이 |

| MANUAL_MOVE | robot_control: move_named(파싱된 이름 — home/front/up/down/watch) | IDLE에서 MANUAL 모드 이동 명령 파싱 | 도달 → IDLE | 모션 실패 → safety_state=FAULT |
| CANCELLING | 진행 중이던 goal에 취소 요청 전달, 완료 대기 | STOP 명령 / 모드 전환 요청 / 안전상태 이상으로 인한 강제 취소 | 취소 확인 → IDLE(STOP) 또는 요청된 새 모드로 진입 | `cancel_timeout_s`(기본 5.0s) 내 취소 확인 안 됨 → safety_state=FAULT |

**`safety_state`(FAULT 등)는 `state`와 별개로 동작**: `/robot/fault` 메시지를
접두어(`PROTECTIVE_STOP:`/`EMERGENCY_STOP:`/`FAULT:`)로 분류하고, 우선순위표에
따라 항상 상향만 한다(조용히 완화되지 않음). 복구는 `RECOVERY_REQUIRED` →
진행 중 goal 취소 확인 → `/robot/recover` 서비스 호출 → 성공 시 `NORMAL` +
`state=IDLE`의 다단계 흐름이며, `recovery_timeout_s`(기본 5.0s)와 응답 세대
카운터로 오래된 비동기 응답을 걸러낸다.

### 3.1 재개(RESUME) 메커니즘 (2026-07-08 추가)

FAULT 진입 직전(`self.state`가 아직 바뀌기 전, `safety_recovery.py`의
`_enter_fault`와 `action_coordinator.py`의 `_handle_action_future_exception`
양쪽에서 공유) 상태를 스냅샷으로 저장한다(`task_flow.py`
`_capture_resume_snapshot`):

- **`continue`** (MOVE_SAFE / APPROACH_HAND / WAIT_PULL) — 파지가 이미 검증된
  뒤이므로 중단된 goal을 그대로 이어서 보낸다(예: WAIT_PULL 중 FAULT였다면
  재개 시 다시 `handover_hold`부터).
- **`retry_pick`** (SERVO_PICK / VERIFY_GRASP) — 그리퍼 상태가 불확실하므로
  이어가지 않고 `release_and_retry`와 동일하게 그리퍼를 강제로 열고 watch로
  돌아가 DETECT_TRACK부터 다시 시작한다(`verify_grasp_max_retries` 재시도
  횟수에 포함되어, 반복되면 결국 safety_state=FAULT로 수동 점검을 요구한다).
- 나머지 상태(IDLE/MOVE_TO_WATCH/DETECT_TRACK/HOME/MANUAL_MOVE/CANCELLING)는
  재개 개념이 없다 — 스냅샷을 남기지 않는다.

복구(`/robot/recover` 성공, safety_state=NORMAL·state=IDLE)가 끝난 뒤 "재개"
명령(`Command.RESUME`, 키워드 "재개")이 오면 `_handle_resume`이 스냅샷에 따라
위 분기로 처리한다. 안전상태가 NORMAL이 아니거나, state가 IDLE이 아니거나,
스냅샷이 없으면 명확한 사유를 `/task/status.detail`로 보고하고 아무 것도 하지
않는다(자동 재시작 금지). 재개는 1회성이라 실행 시작 시점에 스냅샷을 지워
중복 재개를 막고, 정상적으로 한 사이클을 완주(DONE)해도 오래된 스냅샷을
정리한다.

`/task/status`에는 `state`/`safety_state`와 별도로 `resumable`(bool) 필드가
추가됐다 — 스냅샷 존재 ∧ safety_state=NORMAL ∧ state=IDLE일 때만 true이며,
PyQt UI의 재개 버튼(1.1절)이 이 값을 그대로 따라 활성/비활성화된다.

## 4. 커스텀 인터페이스 명세 (`handover_interfaces` 패키지)

### 4.1 msg/ToolTrack.msg

```
std_msgs/Header header          # 관측 시각 — 시간 정합·지연 보상의 기준
string tool_class
geometry_msgs/Pose pose         # base_link 기준, orientation에 yaw 반영.
                                 # depth_valid=false 구간은 z 마지막 유효값 고정, x,y는 RGB 추적으로 갱신
geometry_msgs/Vector3 velocity  # 필터(칼만/알파-베타) 추정 [m/s]
bool approaching                # 하강점 방향으로 이동 중인지
bool depth_valid                # MinZ 이내 근접 등으로 depth가 무효화되었는지
float32 confidence              # 검출 신뢰도(0.0~1.0) — task_manager trigger.min_confidence 게이팅에 사용
```

### 4.2 msg/Detection2D.msg, msg/DetectionArray.msg

object_detection(팀원3) → vision_node(팀원2) 검출 결과 전달 계약. YOLO 검출 자체는
object_detection이 수행하고, vision_node는 이 토픽을 받아 추적(2.3~2.5절)만 수행한다.

```
# Detection2D.msg
string class_name
float32 score
int32 x1
int32 y1
int32 x2
int32 y2

# DetectionArray.msg
std_msgs/Header header           # color 프레임과 동일 stamp
Detection2D[] detections
```

### 4.3 msg/GripperState.msg

```
std_msgs/Header header
float32 width_mm                # 현재 개폐 폭
bool grip_detected              # RG2 그립 감지 플래그
```

### 4.4 srv/SetVisionMode.srv

```
uint8 OFF=0
uint8 TRACK_TOOL=1
uint8 TRACK_HAND=2
uint8 mode
string tool_class               # TRACK_TOOL일 때만 사용
---
bool success
string message
```

### 4.5 action/RobotTask.action

> **실제 현재 정의**(최초 설계 대비 `target_pose`/`measured_payload_kg`/
> `move_pose`/`place_down`이 전부 정리되어 없다 — `robot_control`이 `move_pose`/
> `place_down`을 미지원 task_type으로 명시적으로 거부한다):

```
# --- Goal ---
# task_type: move_named | servo_pick | handover_approach | handover_hold | release_and_retry
string task_type
string named_target             # home / front / up / down / watch / handover_safe 중 하나(move_named 전용)
string tool_class                # servo_pick 전용: 추종·파지 대상 공구 클래스
float32 grasp_width_mm           # servo_pick 전용: 공구별 목표 파지 폭
float32 grasp_force_n            # servo_pick 전용: 파지 힘 (RG2: 3~40N)
# --- Result ---
bool success
float32 final_width_mm          # servo_pick: 폐합 후 실제 폭 (오파지 검출용) — 토크 기반 페이로드 추정은 미구현
bool grip_detected
string message                  # abort 시 사유(발산/추적유실/시야이탈/타임아웃 등) 포함
# --- Feedback ---
string state                    # tracking / descending / closing / lifting 등 (서보 루프 내부 상태)
```

## 6. 대표 시퀀스 — "스패너 갖다줘" 한 사이클

### 6.1 호출 흐름 (인터페이스 수준, 실제 구현 기준)

> `/vision/hand_pose`는 실제로는 vision_node가 `/vision/set_mode`로 모드를
> TRACK_HAND로 전환한 뒤 스트리밍하는 토픽이며, task_manager는 이를 구독하지
> 않는다(3절 표의 "구독" 항목에 없음) — handover_approach 구현(robot_control)이
> 완성되면 그쪽에서 직접 구독할 예정. 12행(vision 모드 전환)은 지금도 실제로
> 동작하는 배선이고, **13행(handover_approach goal)만 현재 TODO 스텁이라 항상
> 실패를 반환한다**(3절 APPROACH_HAND 행 참고).

### 6.2 데이터 수준 시퀀스 (실제 전달 값 예시, 2026-07-07 갱신)

가정: 벨트 속도 0.10 m/s(−x 방향으로 진행 중), 스패너 등록 파지 폭 30mm,
카메라 프로파일 424×240@60fps, 지연 보상 Δt_lat=0.05s. 좌표는 base_link 기준 [m].
**14~18행(VERIFY_GRASP 결과 확인부터 handover_approach goal 발송까지)은 지금도
그대로 일어나는 실제 흐름이다. 19행부터(robot_control이 실제로 접근 이동을
수행하는 부분)만 movel 구현 완료를 가정한 목표 시나리오다 — 2026-07-07 현재는
18행에서 goal을 받은 직후 robot_control이 게이트 통과 후 곧바로 실패를
반환하므로, 실제로는 여기서 safety_state=FAULT로 끝난다(3절 참고).**

| 19 | robot_control 내부 (movel 구현 후) | `/vision/hand_pose`를 한 번만 읽어 TCP→손 방향으로 `stop_distance_m`(5cm) 못 미친 지점까지 movel 이동 |
| 20 | task_manager → robot_control | `task_type: "handover_hold"` |
| 21 | robot_control 내부 | 컴플라이언스 on → 외부 토크 감시 → 당김 힘 임계 초과 감지 → RG2 개방 (이 안에서 완결, 별도 RELEASE goal 없음) |
| 22 | robot_control → task_manager | result `success: true, message: "pull_detected, released"` → task_manager는 곧바로 HOME으로 전이 |
| 23 | task_manager → vision_node / robot_control | `mode: OFF` / `named_target: "home"` |
| 24 | task_manager → UI | `/task/status: {"state": "IDLE", "detail": "DONE tool=spanner"}` |

각 상태 전이마다 task_manager가 `/task/status`에 `state`와 `safety_state`를
함께(예: `{"state": "SERVO_PICK", "safety_state": "NORMAL", "detail": "..."}`)
퍼블리시하여 PyQt UI가 진행 상황을 실시간 표시한다. servo_pick 내부의 세부
반복(필터·추종 루프)은 `RobotTask` Feedback(`state: tracking/descending/closing/lifting`)
으로만 노출되고, task_manager는 이를 그대로 요약해 UI에 전달한다.

## 7. 구현 검증 순서

1. **저해상도 파이프라인 검증**: 정지 공구로 검출률·depth 노이즈·MinZ 실측
2. **시간 정합 검증**: 2.4절의 "로봇 흔들며 정지 공구 추적" 테스트 통과
3. **스트리밍 인터페이스 단독 검증** (완료, 2026-07-07): `probe_speedl_stream.py`로
   비전 없이 실기 검증 — 비-RT `speedl_stream`만으로 RT와 동등하게 부드러운
   추종이 가능함을 확인했고, 명령이 끊겨도 로봇이 스스로 멈추지 않는다는 것과
   `vel=0` 단일 명령으로 충분히 멈춰 유지된다는 것도 실측해 `SpeedlWatchdog`
   설계의 근거로 삼았다(2.2절)
4. **공중 서보**: 하강 없이 수평 추종만 — 컨베이어 모터로 아래 3가지 이동 시나리오를 순서대로 재현하며 확인
   - 4-1. **한방향 등속**: 피드포워드+P 베이스라인 추종 확인 (`w≈1` 유지)
   - 4-2. **긴 주기 왕복**: 방향전환 시 2.5절 innovation 리셋과 그에 따른 `w` 하강·재상승 동작 확인
   - 4-3. **짧은 주기 진동**: `w` 감쇠가 실제로 오차를 줄이는지 — 피드포워드 on(`w` 정상 동작) vs off(`w`=0 고정) 비교 실측으로 `innov_low/high`, `w_alpha` 튜닝
5. **전체 서보 파지**: 하강 포함, 4번의 3가지 시나리오를 그립까지 재검증하며 낮은 게인부터 튜닝

## 8. 역할 분담

### 팀원1 — robot_control 실행 + task_manager

- RobotTask 액션 서버 배선 (goal/result/feedback)
- ~~RT 세션 open/close~~ → **비-RT `speedl_stream` + `SpeedlWatchdog`**로 전환됨(2026-07-07, 2.2절)
- 매 주기 서보 루프(`servo_loop.py`) 호출 → 반환값(`v_cmd`)으로 `speedl_stream` 발행, 상태 기반 폐합/abort 판정
- 그리퍼 제어(RG2) — 폭·힘 설정, `grip_detected` 읽기
- 토크 감시 → `/robot/fault` 퍼블리시
- 정적 모션(move_named / handover_hold) — **move_pose·place_down은 정리되어 미지원**
- `/gripper/state` 퍼블리시
- **task_manager**: `command_parser` 키워드 매칭(**LLM 미사용**), 3절 상태머신(11개 State, FAULT는 별도 safety_state) 구현, RobotTask goal 전송·감독, `/task/status` 퍼블리시

### 팀원2 — vision_tracking + calculate

**vision_node 추적**
- TRACK_TOOL: 프레임 간 추적 → pose·velocity·yaw 추정, `approaching` 판정, `depth_valid` 처리
- TRACK_HAND: 손 검출/추적 → `/vision/hand_pose` 퍼블리시
- `/vision/tool_track` 퍼블리시 (30~60Hz)
- `/vision/set_mode` 서비스 서버 (OFF / TRACK_TOOL / TRACK_HAND)
- `realsense2_camera` launch 구성 + hand-eye 캘리브레이션(flange→camera_link TF 브로드캐스트, 1.6절)

**calculate 모듈 (robot_control 내부, 별도 파일로 분리)**
- 필터(2.5절), 제어법칙(2.3절, `v_cmd = w·v̂_tool + Kp·e`), 피드포워드 가중치 `w`, 파지판정(2.6절), 발산·abort 판정(2.8절)
- **팀원1에게 넘길 인터페이스는 최대한 단순하게 유지**: `입력(ToolTrack, 현재 TCP pose/velocity) → 출력(v_cmd, gripper_close_cmd, abort_reason)` 형태의 함수 하나로 캡슐화하고, 내부 로직(필터·게인·임계값)은 팀원1이 몰라도 호출만으로 쓸 수 있게 한다

### 팀원3 — object_detection + stt

- **object_detection**: YOLO 검출 → `/detection/tool_boxes`(DetectionArray, 4.2절)로 프레임 단위 검출 결과 퍼블리시
- **stt_node**: VAD 발화구간 검출 → `/user_command/text` 퍼블리시

### 공용

- **`operator_gui`(PyQt 데스크톱 앱)**: 세 명 모두 낯선 영역이라 공동 작업
- **멀티 패키지 전체 launch 통합**: 개별 노드 launch는 기존 작업물이 있으므로, 이를 하나로 묶는 통합 launch만 공동으로 정리
- **`handover_interfaces` 패키지**: 4절 명세대로 msg/srv/action 파일 작성 — 설계는 끝나 있어 작업량이 적으므로 필요한 사람이 먼저 만들어 공유
