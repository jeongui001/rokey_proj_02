# 파이프라인 점검

> `docs/초기 설계 시퀀스 다이어그램.md`의 1~49번 항목을 근거로, 실제 하드웨어(로봇팔·그리퍼·카메라·마이크)를
> 전부 연결한 상태에서 처음부터 끝까지 실행해보며 어디까지 작동하는지 확인하기 위한 체크리스트다.
> Phase별로 실행 → 결과 기록 → 문제 있으면 별도로 근본 원인 분석(systematic-debugging).

## 사전 준비

> `handover_system.launch.py`로 한 번에 띄우면 모든 노드 로그가 한 터미널에 섞여서
> 특정 노드(vision_node 등)만 조용히 지켜보기 어렵다. 아래처럼 노드별로 별도 터미널에서
> `ros2 run`으로 띄우면 로그를 분리해서 볼 수 있다.
>
> **주의:** `robot_control.launch.py`는 `hardware_enabled` launch 인자의 기본값이
> `'false'`이고, 이게 `robot_control_params.yaml`의 `hardware_enabled: true`를
> 파라미터 리스트 뒤에서 덮어쓴다. `handover_system.launch.py`는 이 인자를 넘겨주지
> 않으므로 실제로는 dry-run(`hardware_enabled=false`)으로 떠 있었을 가능성이 있다.
> dry-run에서는 `base_link→flange` TF를 방송하지 않아(`robot_control_node.py:501`)
> vision_node의 TF 조회가 항상 실패하고, 그 결과 `_track_tool`이 아예 호출되지 않아
> `/vision/tool_track`이 전혀 퍼블리시되지 않는다 — 지금 막힌 증상과 일치한다.
> 아래 robot_control_node 실행 명령에는 `hardware_enabled:=true`를 명시했다.

각각 별도 터미널에서 (매 터미널마다 워크스페이스 루트로 이동 후 `source install/setup.bash` 먼저):

```bash
cd ~/rokey_proj_02 && source install/setup.bash
```

> `camera_namespace:=""`처럼 빈 문자열 값은 `ros2 launch` CLI가 아예 허용하지 않는다
> (`:=`로 끝나는 인자는 malformed로 거부). 원래 `vision_node.launch.py`가 Python
> 딕셔너리로 넘기던 값이라 CLI로는 흉내낼 수 없다 - 그래서 카메라+정적TF+
> tool_detection_node+vision_node를 원래 launch 파일 그대로 한 터미널에 묶어서 띄운다
> (이 launch 파일이 vision_node 자체도 포함하므로, 별도로 `ros2 run vision_node vision_node`를
> 또 띄우면 vision_node가 두 개 떠서 충돌한다 - 띄우지 말 것).

```bash
# 1) RealSense 카메라 + flange->camera_link 정적 TF + YOLO 검출(tool_detection_node) + 추적/3D복원(vision_node)
# 지금 막힌 지점이라 이 터미널을 집중해서 봐야 함
ros2 launch vision_node vision_node.launch.py
```

```bash
# 2) 로봇 컨트롤러 - hardware_enabled를 명시적으로 true로 지정
ros2 run robot_control robot_control_node --ros-args --params-file ~/rokey_proj_02/install/robot_control/share/robot_control/config/robot_control_params.yaml --params-file ~/rokey_proj_02/install/robot_control/share/robot_control/config/robot_control_calibration_params.yaml -p hardware_enabled:=true
```

```bash
# 3) task_manager
ros2 run task_manager task_manager_node --ros-args --params-file ~/rokey_proj_02/install/task_manager/share/task_manager/config/task_manager_params.yaml
```

```bash
# 4) STT
ros2 run stt_node stt_node
```

```bash
# 5) (선택) operator_gui
OPERATOR_GUI_CAMERA_TOPIC=/vision/debug_image/compressed OPERATOR_GUI_CAMERA_STALE_TIMEOUT_S=2.0 ros2 run operator_gui operator_gui
```

또 다른 터미널에서 상시 모니터링:

```bash
ros2 topic echo /task/status        # 상태 전이 확인 (전 구간 공통)
ros2 topic echo /robot/fault        # 이상 발생 시 즉시 확인
ros2 param get /robot_control hardware_enabled   # dry-run 여부 확인용 (노드 이름은 robot_control, 실행파일명이 robot_control_node인 것과 다름)
```

`/vision/set_mode` 수동 호출 시 mode 값 (`handover_interfaces/srv/SetVisionMode`):
`OFF=0`, `TRACK_TOOL=1`, `TRACK_HAND=2`

| C | `/vision/tool_track` 전혀 안 뜸 (TF 경고조차 없음) | 재현됨 | 카메라 fps를 낮추면 해결된다는 팀원 확인이 있었으나, `424x240x10`은 RealSense가 지원하지 않는 프로파일(ERROR: invalid, 848x480x30으로 자동 복귀 + 리소스 충돌 발생)이었고 `424x240x15`도 아직 검증 전 — fps 가설은 보류하고 `vision_node.launch.py`는 원래 `424x240x60`으로 롤백. 대신 `_on_synced_images`/`_track_tool`에 DEBUG 로그(`vision_node.py`, throttle 1s)를 임시로 추가해 (a) 동기화 콜백 자체가 도는지 (b) mode/tool_class가 맞는지 (c) tracker.update가 None을 반환하는지부터 직접 확인 중. |
