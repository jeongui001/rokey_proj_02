# 공구 전달 로봇 시스템 설계 문서

ROS2 기반 음성 명령 공구 전달 시스템 — 노드 명세, 상태 머신, 인터페이스, 시퀀스
(파지 전략: 비주얼 서보잉 — 하강 중 연속 추적)
멀티 패키지 구조 (Multi-package)로 제작할 것

## 0. 확정된 설계 결정 요약

| 항목 | 결정 |
|---|---|
| 로봇팔 | Doosan M/A 시리즈. 서보 파지 구간은 실시간 제어 인터페이스(RT, 고주기 목표 스트리밍)로 제어. **컨트롤러가 RT 제어를 지원하는지 최우선으로 확인 필요** — 미지원 시 이 설계는 성립하지 않음 |
| 그리퍼 | OnRobot RG2, Compute Box에 Modbus TCP — **robot_control 노드에 통합** (별도 노드 없음) |
| 카메라 | 손목 RealSense(eye-in-hand) 단독. 고정 RGB 카메라 없음. 저해상도·고프레임(424×240@60fps 시작점)으로 MinZ와 비전 지연을 동시에 최소화 |
| 컨베이어 | 등속 양방향 왕복, 제어 없음(노드 없음). 파지는 감시 자세의 RealSense 시야 구간에서만 수행 |
| 파지 전략 | **비주얼 서보잉(PBVS)**: 하강 중에도 RealSense 추적을 계속 소비해 목표를 매 주기 갱신. robot_control 내장 서보 루프가 필터 추정치로 실시간 추종하고, 상태 기반(오차·안정성) 판정으로 폐합. 시간 정합과 지연 보상(Δt_lat)이 핵심 난관 |
| 명령 처리 | 로컬 Whisper STT → LLM API(JSON 구조화 출력) |
| UI | 브라우저 ↔ rosbridge WebSocket |
| 파지/당김 감지 | Doosan 조인트 내장 토크(외부 토크) + RG2 grip_detected + 파지 폭 삼중 확인 |

## 1. 노드 명세

### 1.1 rosbridge_server (기성 패키지)

| | |
|---|---|
| 역할 | 브라우저 UI ↔ ROS2 WebSocket(9090) 중계. 자체 로직 없음 |
| UI → ROS | `/user_command/text` 퍼블리시 (텍스트 명령 입력 경로) |
| ROS → UI | `/task/status`, `/gripper/state`, `/robot/fault` 구독하여 표시 |

### 1.2 stt_node

| | |
|---|---|
| 역할 | 마이크 → VAD 발화 구간 검출 → 로컬 Whisper 추론 → 문장 단위 텍스트 출력 |
| 퍼블리시 | `/user_command/text` (std_msgs/String) |
| 비고 | LLM 호출 없음. 음성/텍스트 입력이 같은 토픽으로 합류하여 이후 경로 단일화 |

### 1.3 task_manager — 명령 해석과 감독 주체

| | |
|---|---|
| 역할 | LLM 명령 해석, 상태 머신 실행, SERVO_PICK 시작·감독(발산/타임아웃 판단), 모든 재시도/실패 판단 |
| 구독 | `/user_command/text`, `/vision/tool_track`(DETECT_TRACK 트리거 판정용), `/robot/fault` |
| 퍼블리시 | `/task/status` (상태 전이·진행 상황, JSON 문자열) |
| 서비스 클라 | `/vision/set_mode` |
| 액션 클라 | `RobotTask` (robot_control) |
| 외부 API | LLM API 호출 → `{"tool": "...", "action": "..."}` 파싱, 스키마 검증·재시도 |
| 핵심 로직 | ① 완화된 트리거 판정(목표 공구가 시야 내 + 접근 중이면 충분 — 체류 시간·전환 지점 이격 조건 불필요, 서보가 따라가므로) ② `servo_pick` goal 1회 전송(tool_class, grasp_width_mm, grasp_force_n) 후 감독만 수행 — 좌표·시각 계산은 robot_control로 이관 ③ abort 수신 시 사유별 재시도/보고 판단 |

### 1.4 vision_node — RealSense 전담

| | |
|---|---|
| 역할 | 저해상도·고프레임 파이프라인. 모드에 따라 공구 추적 또는 손 추적. 검출 → 3D 복원 → base_link 변환 → 스트리밍 |
| 구독 | `/camera/color/image_raw`, `/camera/aligned_depth_to_color/image_raw`, `camera_info`, TF |
| 퍼블리시 | `/vision/tool_track` (ToolTrack, 30~60Hz), `/vision/hand_pose` (PoseStamped) |
| 서비스 서버 | `/vision/set_mode` — `OFF` / `TRACK_TOOL(공구클래스)` / `TRACK_HAND` |
| TRACK_TOOL | 저해상도 입력(YOLO 416 또는 320)으로 검출 + 프레임 간 추적해 pose·velocity·yaw 추정, `approaching` 판정. depth가 MinZ 이내 근접 등으로 무효화되면 `depth_valid=false`로 표시하고, 이 구간에서도 RGB 추적으로 x·y는 계속 갱신(z는 마지막 유효값 고정) |
| TRACK_HAND | 손 검출(MediaPipe 등) → 전달 위치 미세 조정용 손 자세 |
| 비고 | 모든 메시지에 관측 시각(header.stamp) 필수 — 시간 정합·지연 보상의 기준. 3D 복원용 TF는 "지금"이 아니라 "이미지가 촬영된 시각"의 flange pose로 조회(`tf_buffer.lookup_transform(..., 이미지 header.stamp)`). 비전 PC와 로봇 제어 PC가 분리되어 있으면 chrony 시계 동기화 필수 |

### 1.5 robot_control — 실행자 + 서보 제어 루프 내장

| | |
|---|---|
| 역할 | RobotTask 액션 서버. Doosan 모션 서비스(정적 이동) + 실시간 제어 인터페이스(서보 파지 전용) 호출 + RG2 그리퍼 제어(Modbus TCP 내장) + 토크 감시 |
| 액션 서버 | `RobotTask` — move_named / move_pose / servo_pick / handover_hold / place_down |
| 구독 | Doosan 드라이버 로봇 상태(외부 토크 포함, 최대 주기로 발행 요청), `/vision/tool_track` (servo_pick 진행 중 직접 구독) |
| 퍼블리시 | `/gripper/state` (GripperState, 상시), `/robot/fault` (protective stop 등) |
| servo_pick | goal 수신 시 실시간 제어(RT) 세션 오픈 → `/vision/tool_track`을 직접 구독해 필터(칼만/알파-베타)로 위치·속도 추정 → PBVS 제어 법칙으로 매 주기 목표 추종(2절) → 상태 기반 판정 성립 시 RG2 폐합(공구별 폭·힘) → 즉시 들어올림 → 토크 변화로 페이로드 추정 → RT 세션 종료 후 서비스 모션 모드로 복귀 → result 반환 |
| handover_hold | 컴플라이언스 모드 on + 외부 토크 임계 감시 → 당김 감지 시 RG2 개방, result 반환 |
| 내부 구조 | RG2 제어는 `rg2_client.py` 클래스로 분리(추후 노드 분리 대비). 서보 제어 루프는 별도 모듈로 분리 권장 |

### 1.6 realsense2_camera (기성 패키지)

Depth/Color 424×240 @ 60fps(시작점), High Density 또는 Short Range 프리셋으로 근거리 fill rate 우선.
launch에서 hand-eye 캘리브레이션 결과인 `flange → camera_link` static TF를 함께 브로드캐스트.

## 2. 서보 제어 루프 설계 (robot_control 내부)

task_manager가 좌표를 조립하고 robot_control은 순수 실행만 한다는 원칙에 이 구간만 예외가 있다.
서보 루프는 지연에 민감해 task_manager를 경유할 수 없으므로, servo_pick 동안에는
robot_control이 `/vision/tool_track`을 **직접 구독**해 제어 루프를 돌린다.

```
vision ────────────────────────▶ robot_control (서보 루프, 매 주기)
              task_manager는 시작/중단/감독만 (goal/abort/result)
```

### 2.1 카메라 설정과 지연 예산

관건은 **비전 루프 지연**과 **블라인드 구간**이며, 해상도를 낮추면 둘 다 개선된다.

1. **지연 감소**: 프레임 전송·YOLO 추론 시간이 해상도에 비례해 줄어 30~60Hz 갱신이 현실화된다.
2. **MinZ 감소**: RealSense 스테레오 최소 심도 거리(MinZ)는 대략 가로 해상도에 비례한다.
   D435 기준 1280×720에서 약 28cm → 848×480 약 18~20cm → 424×240 약 10cm 수준
   (기종·프리셋별 상이, **반드시 실측**). 저해상도일수록 depth가 유효한 채로
   공구에 더 가까이 접근할 수 있어 블라인드 종단 구간이 짧아진다.

해상도 저하의 대가로 depth 노이즈와 검출 정밀도가 나빠지므로 필터(2.5절)가 필수가 된다.

| 항목 | 값 | 근거 |
|---|---|---|
| Depth 해상도/프레임 | 424×240 @ 60fps | MinZ 최소화 + 지연 최소화 |
| Color 해상도/프레임 | 424×240 @ 60fps | depth와 정렬 비용 최소화 |
| YOLO 입력 | 416 또는 320 | 추론 10~20ms(GPU) 목표 |
| 프리셋 | High Density 또는 Short Range | 근거리 fill rate 우선 |

지연 예산 (목표: 관측→명령 반영 70ms 이내):

| 구간 | 예상 |
|---|---|
| 노출 + 카메라 파이프라인 + USB | 20~35ms |
| YOLO 추론 + 3D 복원 + 필터 | 10~20ms |
| DDS 전달 + 서보 루프 반영 | 2~5ms |
| **합계 (lookahead 예측에 사용, Δt_lat)** | **약 40~60ms** |

벨트 0.10m/s 기준 40~60ms는 4~6mm에 해당하므로 보상 없이는 그대로 파지 오차가 된다.

### 2.2 Doosan 실시간 제어 인터페이스

서비스 모션(movel: 목표 주고 완료 대기)은 도중 갱신이 불가하므로, Doosan의
실시간 제어 모드(고주기 목표 스트리밍, 컨트롤러·드라이버 버전에 따라
servo/speed 계열 RT 명령)로 전환해야 한다. **사용 중인 컨트롤러 버전이 RT 제어를
지원하는지, 제어 주기와 명령 형식이 무엇인지 드라이버 문서로 최우선 확인할 것.**

robot_control은 servo_pick 진입 시 RT 세션을 열고, 종료(성공/abort/fault) 시
반드시 닫은 뒤 서비스 모션 모드로 복귀한다.

### 2.3 제어 법칙 (PBVS, base_link 기준)

루프 주기: 비전 갱신(30~60Hz)과 RT 명령 주기가 다르므로, 비전은 필터 갱신에만
쓰고 RT 명령은 필터의 예측값을 자체 주기로 내보낸다(비전 프레임 사이에는 외삽).

```
목표점  p_ref(t) = p̂_tool(t + Δt_lat) + offset_approach
오차    e = p_ref − p_tcp
명령    v_cmd = clamp(Kp · e, v_max)          # 수평(x,y) 성분
하강    z는 별도 프로파일: 수평 오차 |e_xy| < ε_descend 인 동안만 일정 속도로 하강,
        초과하면 하강 일시 정지(수평 정렬 우선)
yaw     동일하게 P 제어, 각속도 제한
```

- `p̂_tool(·)`: 필터(2.5절)의 상태로부터 `위치 + 속도 × (Δt_lat + 루프 반주기)` 외삽
- Kp는 낮게 시작(0.8~1.5 /s), v_max는 벨트 속도의 2~3배면 충분

### 2.4 시간 정합 — 이 방식의 최대 함정

eye-in-hand로 팔이 움직이는 중에는, 3D 복원에 쓰는 TF 변환을 **"지금"이 아니라
"이미지가 촬영된 시각"의 flange pose**로 해야 한다.

- vision_node: `tf_buffer.lookup_transform(base, camera, 이미지 header.stamp)` 사용.
  Doosan 드라이버의 로봇 상태 퍼블리시 주기가 낮으면 TF 보간 오차가 커지므로
  상태 퍼블리시 주기를 최대로 올릴 것
- 비전 PC와 로봇 제어 PC가 분리되어 있으면 chrony 시계 동기화 필수
  (수십 ms 어긋남 = 수 mm 오차 직결)
- 검증법: 로봇을 일부러 흔들면서 정지한 공구를 추적 → base_link 좌표가
  흔들리지 않으면 정합 성공

### 2.5 필터

base_link 좌표의 공구 위치에 등속 모델 칼만 필터(또는 알파-베타 필터)를 적용.
상태 = [x, y, z, vx, vy], 저해상도 depth의 노이즈를 흡수하고 velocity를 안정적으로
추정하는 것이 목적. 벨트 **방향 전환은 등속 모델에 대한 급격한 위반**이므로,
innovation(예측-관측 잔차)이 임계를 넘으면 속도 공분산을 리셋해 빠르게 재수렴시킨다.

### 2.6 파지 판정과 폐합

정지 시각 예약이 아니라 **상태 기반 판정**으로 이루어진다:

```
if |e_xy| < ε_grasp (예 5mm) 가 N주기(예 5) 연속
   and z_gap < z_close (예 2cm)
   and 필터 공분산 < 임계:
       그리퍼 폐합 (RG2: 공구별 폭·힘)
```

### 2.7 종단 블라인드 구간 처리

MinZ를 10cm 수준까지 낮춰도 마지막 수 cm + 그리퍼 자체 가림은 남는다:

1. depth 무효 진입 후: z는 마지막 유효값 고정, x·y는 RGB 추적으로 계속 보정
   (`depth_valid=false` 구간)
2. RGB마저 가려지면: 필터의 마지막 속도로 짧은 탄도(ballistic) 종단 —
   구간이 1~3cm로 짧아 오차가 무시할 수준

### 2.8 발산·중단(abort) 조건

다음 중 하나면 서보를 중단하고 감시 자세로 상승 복귀 → task_manager에 abort 반환
→ DETECT_TRACK 재진입(토크 이상은 즉시 FAULT):

- |e_xy|가 M주기 연속 증가 (발산)
- 추적 유실(confidence 저하·검출 끊김)이 T_lost(예 0.3s) 초과
- 공구가 방향 전환하여 시야 이탈 예상
- 서보 시간이 총 타임아웃(예 5s) 초과
- 외부 토크 이상(충돌) → 즉시 정지 + FAULT

## 3. 상태 머신 및 상태 × 노드 매트릭스

전체 흐름:

```
IDLE → PARSING → MOVE_TO_WATCH → DETECT_TRACK → SERVO_PICK → VERIFY_GRASP
     → MOVE_SAFE → TRACK_HAND → WAIT_PULL → RELEASE → HOME → IDLE
(모든 상태에서 /robot/fault 수신 시 → FAULT, 수동 리셋으로만 IDLE 복귀)
```

| 상태 | 활성 동작 (노드별) | 진입 조건 | 정상 탈출 | 실패 전이 |
|---|---|---|---|---|
| IDLE | task_manager: `/user_command/text` 대기. vision: OFF | 시스템 기동 / HOME 완료 | 명령 수신 → PARSING | — |
| PARSING | task_manager: LLM API 호출, JSON 스키마 검증 | 명령 수신 | 파싱 성공 → MOVE_TO_WATCH | 파싱 실패·미지 공구 → UI로 되묻기 → IDLE |
| MOVE_TO_WATCH | robot_control: move_named(watch). task_manager: vision을 TRACK_TOOL(목표 클래스)로 전환 | PARSING 성공 | 이동 완료 → DETECT_TRACK | 모션 실패 → FAULT |
| DETECT_TRACK | vision: 공구 추적 스트리밍(저해상도·고프레임). task_manager: 트리거 조건 감시(시야 내 + 접근 중이면 충분 — 체류 시간·전환 지점 이격 불필요). **로봇팔 정지** | 감시 자세 도달 | 트리거 성립 → SERVO_PICK | 왕복 2~3주기 미검출 → "벨트에 없음" 보고 → IDLE |
| SERVO_PICK | robot_control: RT 세션 오픈, `/vision/tool_track` 직접 구독해 PBVS 서보 루프로 추종·폐합·들어올림(2절). task_manager: goal 1회 전송 후 감독만 | 트리거 성립 | 파지·들어올림 완료 → VERIFY_GRASP | 발산·추적유실·시야이탈·타임아웃 → abort → DETECT_TRACK 복귀. 토크 이상 → 즉시 FAULT |
| VERIFY_GRASP | robot_control: 토크 페이로드 + grip_detected + 파지 폭 삼중 확인 (result로 반환), task_manager: 판정 | 들어올림 완료 | 3중 확인 통과 → MOVE_SAFE | 실패 → RG2 개방 후 DETECT_TRACK, 최대 2회 초과 시 보고 → IDLE |
| MOVE_SAFE | robot_control: move_named(safe), 속도 제한 적용 | 파지 확인 | 도달 → TRACK_HAND | 모션 실패 → FAULT |
| TRACK_HAND | vision: TRACK_HAND 모드 → `/vision/hand_pose`. task_manager: 전달 자세 미세 조정 goal 전송 | 안전 자세 도달 | 손 근처 도달 → WAIT_PULL | 손 미검출 타임아웃 → 기본 전달 자세로 WAIT_PULL |
| WAIT_PULL | robot_control: handover_hold(컴플라이언스 on, 토크 임계 감시) | 전달 자세 도달 | 당김 감지 → RELEASE | 타임아웃(예 60s) → place_down(지정 위치) → HOME |
| RELEASE | robot_control: RG2 개방 + 힘 감소, 개방 확인 | 당김 감지 | 개방 완료 → HOME | — |
| HOME | robot_control: move_named(home). task_manager: 완료 보고, vision OFF | 개방/내려놓기 완료 | 도달 → IDLE | 모션 실패 → FAULT |
| FAULT | robot_control: 정지 유지. task_manager: UI에 오류 표시, 수동 리셋 대기 | protective stop / 액션 타임아웃 / 토크 이상 | UI 리셋 → IDLE | — |

## 4. 커스텀 인터페이스 명세 (`handover_interfaces` 패키지)

### 4.1 msg/ToolTrack.msg

```
std_msgs/Header header          # 관측 시각 — 시간 정합·지연 보상의 기준
string tool_class
geometry_msgs/Pose pose         # base_link 기준, orientation에 yaw 반영.
                                 # depth_valid=false 구간은 z 마지막 유효값 고정, x,y는 RGB 추적으로 갱신
geometry_msgs/Vector3 velocity  # 필터(칼만/알파-베타) 추정 [m/s]
bool approaching                # 하강점 방향으로 이동 중인지
bool depth_valid                # MinZ 이내 근접 등으로 depth가 무효화되었는지
float32 confidence
```

### 4.2 msg/GripperState.msg

```
std_msgs/Header header
float32 width_mm                # 현재 개폐 폭
bool grip_detected              # RG2 그립 감지 플래그
```

### 4.3 srv/SetVisionMode.srv

```
uint8 OFF=0
uint8 TRACK_TOOL=1
uint8 TRACK_HAND=2
uint8 mode
string tool_class               # TRACK_TOOL일 때만 사용
---
bool success
string message
```

### 4.4 action/RobotTask.action

```
# --- Goal ---
string task_type                # move_named | move_pose | servo_pick | handover_hold | place_down
string named_target             # watch / safe / home / place_down 자세 이름
geometry_msgs/PoseStamped target_pose   # base_link 기준 (move_pose 전용, servo_pick은 무시)
string tool_class               # servo_pick 전용: 추종·파지 대상 공구 클래스
float32 grasp_width_mm          # servo_pick 전용: 공구별 목표 파지 폭
float32 grasp_force_n           # servo_pick 전용: 파지 힘 (RG2: 3~40N)
# --- Result ---
bool success
float32 measured_payload_kg     # servo_pick: 들어올림 후 토크 기반 추정 무게
float32 final_width_mm          # servo_pick: 폐합 후 실제 폭 (오파지 검출용)
bool grip_detected
string message                  # abort 시 사유(발산/추적유실/시야이탈/타임아웃/토크이상) 포함
# --- Feedback ---
string state                    # tracking / descending / closing / lifting / holding / aborting ...
```

## 6. 대표 시퀀스 — "스패너 갖다줘" 한 사이클

### 6.1 호출 흐름 (인터페이스 수준)

| # | 상태 | 방향 | 인터페이스 | 내용 |
|---|---|---|---|---|
| 1 | IDLE | 사용자 → stt_node | 마이크(음성) | "스패너 갖다줘" |
| 2 | IDLE | stt_node → task_manager | `/user_command/text` | 인식된 문장 |
| 3 | PARSING | task_manager ↔ LLM API | HTTPS | 명령 해석 요청/응답 |
| 4 | MOVE_TO_WATCH | task_manager → robot_control | RobotTask goal | 감시 자세로 이동 |
| 5 | MOVE_TO_WATCH | task_manager → vision_node | `/vision/set_mode` | 공구 추적 모드 시작 |
| 6 | DETECT_TRACK | vision_node → task_manager | `/vision/tool_track` (30~60Hz 스트림) | 위치·속도·접근 여부·depth_valid |
| 7 | SERVO_PICK | task_manager → robot_control | RobotTask goal (servo_pick) | tool_class·grasp_width_mm·grasp_force_n — 1회 전송 |
| 8 | SERVO_PICK | vision_node → robot_control | `/vision/tool_track` (직접 구독, task_manager 미경유) | 서보 루프 입력 스트림 |
| 9 | SERVO_PICK | robot_control 내부 | PBVS 서보 루프 | 필터 추정 → 추종 → 상태 기반 폐합 → 들어올림 |
| 10 | VERIFY_GRASP | robot_control → task_manager | RobotTask result | 무게·폭·그립 감지 |
| 11 | MOVE_SAFE | task_manager → robot_control | RobotTask goal | 안전 자세 이동(속도 제한) |
| 12 | TRACK_HAND | task_manager → vision_node | `/vision/set_mode` | 손 추적 모드 전환 |
| 13 | TRACK_HAND | vision_node → task_manager | `/vision/hand_pose` | 사용자 손 위치 |
| 14 | TRACK_HAND | task_manager → robot_control | RobotTask goal | 전달 자세 미세 조정 |
| 15 | WAIT_PULL | task_manager → robot_control | RobotTask goal (handover_hold) | 컴플라이언스 + 당김 감시 |
| 16 | RELEASE | robot_control → task_manager | RobotTask result | 당김 감지·개방 완료 |
| 17 | HOME | task_manager → vision_node / robot_control | set_mode(OFF) / goal(home) | 종료 정리 |
| 18 | IDLE | task_manager → 브라우저 UI | `/task/status` (rosbridge 경유) | 완료 보고 |

### 6.2 데이터 수준 시퀀스 (실제 전달 값 예시)

가정: 벨트 속도 0.10 m/s(−x 방향으로 진행 중), 스패너 등록 무게 0.30kg·파지 폭 30mm,
카메라 프로파일 424×240@60fps, 지연 보상 Δt_lat=0.05s. 좌표는 base_link 기준 [m].

| # | 방향 | 전달 데이터 (예시 값) |
|---|---|---|
| 1 | 사용자 → stt_node | (음성) "스패너 갖다줘" |
| 2 | stt_node → task_manager | `data: "스패너 갖다줘"` |
| 3 | task_manager → LLM | 시스템 프롬프트(공구 목록·JSON 스키마) + 사용자 문장 |
| 4 | LLM → task_manager | `{"tool": "spanner", "action": "handover"}` |
| 5 | task_manager → robot_control | `task_type: "move_named", named_target: "watch"` |
| 6 | robot_control → task_manager | result `success: true` |
| 7 | task_manager → vision_node | `mode: TRACK_TOOL, tool_class: "spanner"` |
| 8 | vision_node → task_manager (스트림) | `pose: (0.82, 0.15, 0.045, yaw 35°), velocity: (−0.100, 0, 0), approaching: true, depth_valid: true, confidence: 0.93, stamp: t₀` |
| 9 | task_manager 내부 | 완화된 트리거 판정(시야 내 + approaching) 성립 |
| 10 | task_manager → robot_control | `task_type: "servo_pick", tool_class: "spanner", grasp_width_mm: 30, grasp_force_n: 20` (1회 전송) |
| 11 | robot_control 내부 | RT 세션 오픈, `/vision/tool_track` 직접 구독 시작 |
| 12 | vision_node → robot_control (직접, 스트림) | `pose: (0.71, 0.15, 0.045), velocity: (−0.098, 0, 0), depth_valid: true, stamp: t₁` |
| 13 | robot_control 내부 | 필터 갱신 → `p_ref = p̂_tool(t+Δt_lat) + offset` → `e_xy` 계산 → `v_cmd = Kp·e` 수평 추종. `|e_xy| > ε_descend`라 하강 대기 |
| 14 | vision_node → robot_control (직접, 스트림) | `pose: (0.65, 0.15, 0.045), velocity: (−0.099, 0, 0), depth_valid: true, stamp: t₂` |
| 15 | robot_control 내부 | `|e_xy| < ε_descend(15mm)` → 하강 시작(0.10m/s) |
| 16 | vision_node → robot_control (직접, 스트림) | `pose: (0.645, 0.152, 0.045), depth_valid: false`(MinZ 근접 — z는 직전 유효값 0.045로 고정, x·y만 RGB 추적으로 갱신), `stamp: t₃` |
| 17 | robot_control 내부 | 종단 블라인드 구간 진입 — RGB 추적으로 x·y 계속 보정 |
| 18 | robot_control 내부 | `|e_xy| < ε_grasp(5mm)` 5주기 연속 ∧ `z_gap < 2cm` ∧ 필터 공분산 < 임계 → 폐합 판정 → RG2 폐합(30mm, 20N) → 즉시 들어올림 → 외부 토크로 페이로드 추정 → RT 세션 종료 |
| 19 | robot_control → task_manager | result `success: true, measured_payload_kg: 0.31, final_width_mm: 29.4, grip_detected: true` |
| 20 | task_manager 내부 | 삼중 확인: 0.31 ≈ 0.30kg ∧ 29.4 ≈ 30mm ∧ grip_detected → 통과 |
| 21 | task_manager → robot_control | `task_type: "move_named", named_target: "safe"` (속도 제한 프로파일) |
| 22 | task_manager → vision_node | `mode: TRACK_HAND` |
| 23 | vision_node → task_manager | `hand_pose: (0.35, −0.42, 0.30)` |
| 24 | task_manager → robot_control | `task_type: "move_pose", target_pose: (0.35, −0.42, 0.38)` — 손 위 8cm |
| 25 | task_manager → robot_control | `task_type: "handover_hold"` |
| 26 | robot_control 내부 | 컴플라이언스 on → 외부 토크 감시 → 당김 힘 임계(예: 4N) 초과 감지 → RG2 개방 |
| 27 | robot_control → task_manager | result `success: true, message: "pull_detected, released"` |
| 28 | task_manager → vision_node / robot_control | `mode: OFF` / `named_target: "home"` |
| 29 | task_manager → UI | `/task/status: {"state": "DONE", "tool": "spanner", "payload_kg": 0.31}` |

각 상태 전이마다 task_manager가 `/task/status`에
`{"state": "SERVO_PICK", "detail": "비주얼 서보잉 파지 중"}` 형태로 퍼블리시하여
브라우저 UI가 진행 상황을 실시간 표시한다. servo_pick 내부의 세부 반복(필터·추종 루프)은
`RobotTask` Feedback(`state: tracking/descending/closing/lifting`)으로만 노출되고,
task_manager는 이를 그대로 요약해 UI에 전달한다.

## 7. 구현 검증 순서

1. **저해상도 파이프라인 검증**: 정지 공구로 검출률·depth 노이즈·MinZ 실측
2. **시간 정합 검증**: 2.4절의 "로봇 흔들며 정지 공구 추적" 테스트 통과
3. **RT 인터페이스 단독 검증**: 비전 없이 정현파 목표 추종으로 제어 주기·지연 확인
4. **공중 서보**: 하강 없이 수평 추종만 — 움직이는 공구 위 상공을 따라가는지
5. **전체 서보 파지**: 하강 포함, 낮은 게인부터 튜닝
